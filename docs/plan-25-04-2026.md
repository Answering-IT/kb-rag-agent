1. KB Remining Definition

Metadata for tenant:
- All files in knowledge base should have the tenantId (OrganizationId) on its metadata
- KB Code documentation should have tenantId all, so people can chat about the software (support agent)

Necesito encontrar una forma de brindar accesos a través de la metadata, esto es un poco distinto a los PII pero podemos ver si a traves de eso se puede implementar.

El requerimiento es poder acceder o no a la información, siendo capaces de haber cargado los limites de accesos en los documentos subidos.

Por ejemplo:

Dado que ya existen documentos previamente cargados:

info_general_de_la_empresa_tenant1.pdf, metadata { tenantId: 1, roles: [*] }
/tenant_1/projects/1/carta especifica.pdf { tenantId: 1, roles: [9-supervisor], projectId: 1, users: [*], fileName: ....  } 
/tenant_1/projects/1/carta reumen.pdf { tenantId: 1, roles: [9-supervisor,10-asesor], projectId: 1, users: [*], fileName: .... }
/tenant_1/projects/1/respuesta colaboracion.pdf { tenantId: 1, roles: [9-supervisor,10-asesor], projectId: 1, users: [xyz,ijk], fileName: .... }

info_general_de_la_empresa_tenant2.pdf, metadata { tenantId: 2, roles: [*]}
/tenant_2/projects/1/carta especifica.pdf { tenantId: 1, roles: [9-supervisor], projectId: 1, users: [*], fileName: ....  } 
/tenant_2/projects/1/carta reumen.pdf { tenantId: 1, roles: [9-supervisor,10-asesor], projectId: 1, users: [*], fileName: .... }
/tenant_2/projects/1/respuesta colaboracion.pdf { tenantId: 1, roles: [9-supervisor,10-asesor], projectId: 1, users: [xyz,ijk], fileName: .... }

Escenario 1:

El usuario xyz, 
- pertenece a los tenant {id:1 code: colp} y {id:2 code: ac}
- el usuario tiene roles 9-supervisor y 10-asesor para el tenan 1 pero solo tiene rol 10-asesor para el tenant2
- El usuario está logueado en el tenant id 1 con rol 10-asesor
- El usuario pregunta por información, los filtros deben aplicarse por tenantId, role, projectId, y users en ese orden de prioridad dependiendo de lo que se envíe
- Las coincidencias deberían ser información dentro de los ficheros con esos accesos
- Luego entran los PII filters para ver que información puede dar

Mas escenarios añadelos tu.


No quiero que sea demasiado excluyente que no pueda nisiquiera probarse, pero si quiero que en base a varias variables poder tener la flexibilidad de no proveer información que no se debe, optimizar la consulta y proveer información de forma segura.

Quiero que la solución sea simple y facil de mantener en el tiempo para multiples tenants y demás.

Las variables se deberían poder pasar por el API gateway, algunas como headers otras en el request.

2. Agent need...

El agente actual solo responde preguntas, pero necesitamos poder llamar tools usando los Gateway https://docs.aws.amazon.com/AWSCloudFormation/latest/TemplateReference/aws-resource-bedrockagentcore-gateway.html, por ejemplo:
- get_project_info_tool, consultar la información del proyecto por el projectId, através de http request, arn arn:aws:ecs:us-east-1:708819485463:service/dev-answering-colpensiones-ecs-cluster-app/dev-answering-colpensiones-app-fg-srv-jariapp  GET "/organization/1/projects/{projectId}"
- consultar otro tipo de cosas y ejecutar acciones

La implementación debería ser escalable a poder añader más tools disponibles

El agente también debería tener memoria https://docs.aws.amazon.com/AWSCloudFormation/latest/TemplateReference/aws-resource-bedrockagentcore-memory.html respecto a la conversación que está teniendo con el usuario, usemos agent core memory para esto check, https://docs.aws.amazon.com/cdk/api/v2/docs/aws-cdk-lib.aws_bedrockagentcore-readme.html or https://docs.aws.amazon.com/AWSCloudFormation/latest/TemplateReference/AWS_BedrockAgentCore.html

El actual api gateway debería permitir proveer la respuesta al cliente en forma de streaming para garantizar buena experiencia.

Dividamos esto en dos fases.

Fase 1 todo con el Agente Actual
Fase 2 crear un RAG Agent v2 con aws-cdk-lib.aws_bedrockagentcore en un nuevo stack que se conecte a toda nuestra infra también, esto nos da la posibilidad de tener varios agentes conectados al mismo KB

Nice to have, tendremos más agentes en este ecosistema asi que vendría bien mantener el agent stack pero generalizarlo en cdk para poder crear otros agentes en el futuro.
