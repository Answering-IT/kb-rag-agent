OCR LAMBDA WORKFLOW
===================

Step 1: S3 Upload Event
- EventBridge detects new document in documents/ prefix
- Triggers OCR Lambda

Step 2: Textract Job Start
- OCR Lambda calls StartDocumentAnalysis
- Features: TABLES, FORMS
- SNS topic for completion notification
- Async processing (can take minutes for large docs)

Step 3: Textract Completion
- Textract publishes to SNS topic
- OCR Lambda receives SNS notification
- Lambda calls GetDocumentAnalysis

Step 4: Text Extraction
- OCR Lambda extracts all text blocks
- Chunks text (current: sends to SQS)
- PROPOSED: Write extracted text to documents/processed/

Step 5: Vectorization
- CURRENT: Embedder Lambda generates embeddings
- PROPOSED: Bedrock KB Sync handles everything

ARCHITECTURAL DECISION: OCR SIMPLIFICATION
==========================================

Current State:
- OCR Lambda extracts text
- OCR Lambda chunks text
- OCR Lambda sends chunks to SQS
- Embedder Lambda reads SQS
- Embedder Lambda generates embeddings
- Embedder Lambda writes to vectorsBucket (regular S3)
- Bedrock KB Sync ALSO processes original documents
- Bedrock generates embeddings AGAIN
- Bedrock writes to VectorBucket (AWS::S3Vectors)

Proposed Simplified State:
- OCR Lambda extracts text ONLY
- OCR Lambda writes to documents/processed/
- Bedrock KB Sync processes BOTH:
  - Original documents (documents/)
  - OCR-processed text (documents/processed/)
- Bedrock handles chunking + embedding + indexing
- Single pipeline, no duplication

Benefits:
- 50% cost reduction (no duplicate embeddings)
- Simpler architecture (fewer components)
- Easier debugging (single pipeline)
- Faster processing (Bedrock optimized)
